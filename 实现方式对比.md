# 法律合同中译英且保留格式：实现方式对比

目标：中文法律合同转英文，在联系上下文的前提下尽量保留 Word 原格式，包括高亮、粗体、下划线、表格、页眉页脚。

## 方案一：DOCX run 级翻译，本地工具接 OpenAI / DeepSeek API

这是本文件夹当前默认实现的方案。

做法：

1. 选择 OpenAI 或 DeepSeek，并输入对应 API Key。
2. 读取 `.docx` 文件。
3. 提取整份合同的术语表、当事方、定义词、反复出现的法律短语。
4. 默认按段落并发翻译，但每个段落仍按 Word run 返回译文。
5. 如果某个 run 漏翻或仍有中文，只补译该 run。
6. 把译文写回对应 Word run。run 是 Word 内部用于保存粗体、下划线、高亮等格式的最小文本片段。
7. 保存成用户指定位置的新 `.docx`。

优点：

- 可以做成最简单的“选择 API + 输入 API Key + 选择模式 + 拖文件 + 选择输出位置 + 导出”。
- 并发 run 模式能在保留格式的前提下提升速度。
- 粗体、下划线、高亮等 run 格式保留能力较好。
- 可加入法律翻译提示词、术语表、上下文记忆，法律英语质量比普通机器翻译更可控。
- 数据不需要上传给除 OpenAI API 之外的第三方文档平台。

缺点：

- 如果英文语序和中文 run 切分差异很大，格式位置会严格跟随原 run，这是“格式必须完全一样”的取舍。
- 如果一句话内部有很多零碎格式，英文语序变化后，格式位置可能不是 100% 符合人工排版习惯。
- 文本框、形状、批注、修订痕迹等复杂 Word 对象处理有限。
- 需要 API 费用；OpenAI 和 DeepSeek 的翻译质量、速度、价格会不同。

适合：

- Word 合同、协议、备忘录、授权书。
- 重点是“原版式尽量不动 + 法律英语质量可控”。

## 方案二：DOCX 转 XLIFF / HTML 标签，再翻译后还原

做法：

1. 把 DOCX 转成带内联标签的翻译交换格式，例如 XLIFF。
2. 把格式标签保护起来，只翻译文本。
3. 翻译完成后还原成 DOCX。

优点：

- 是翻译行业常见思路。
- 对复杂内联格式更专业。
- 适合术语库、翻译记忆库、多人审校流程。

缺点：

- 工程复杂度明显更高。
- 对小白用户不友好。
- 需要更多格式转换和质检逻辑。

适合：

- 翻译公司、长期批量合同翻译、需要翻译记忆库和审校流程的团队。

## 方案三：云文档翻译 API，例如 Azure / Google 文档翻译

做法：

1. 上传 DOCX。
2. 云服务直接返回译文 DOCX。

优点：

- 接入快。
- 格式保持通常不错。
- 批量文档处理方便。

缺点：

- 法律英语风格和上下文控制较弱。
- 术语一致性、定义词处理、条款级法律表达不一定稳定。
- 需要额外云账号和权限配置。

适合：

- 对法律精度要求不高的大批量初译。

## 方案四：PDF / 图片 OCR 后翻译再重排

做法：

1. OCR 识别扫描件或 PDF。
2. 翻译。
3. 重新生成 Word 或 PDF。

优点：

- 可以处理扫描件。

缺点：

- “格式不变”最难保证。
- OCR 错字会直接影响法律含义。
- 表格、页眉、印章、脚注、双栏版式容易错位。

适合：

- 没有 Word 原稿时的补救流程，不建议作为首选。

## 推荐落地路线

第一阶段：使用当前本地 DOCX 工具。

- 做到 OpenAI/DeepSeek 可选、API Key 输入、拖入 `.docx`、选择输出文件、自动导出。
- 适合个人或小团队马上使用。

第二阶段：增加人工审校辅助。

- 导出双语对照表。
- 标记低置信度段落。
- 生成术语表。

第三阶段：走专业翻译生产线。

- 引入 XLIFF。
- 引入术语库和翻译记忆库。
- 支持批量任务和审校记录。

## 当前工具的核心取舍

为了让完全不懂技术的人也能使用，当前版本选择了“本地 GUI + OpenAI/DeepSeek API + DOCX 标签批量翻译 + run 级保格式还原”。它不是完整翻译生产线级别的 XLIFF/TM 系统，但比逐段 API 调用快很多，也更容易交付给普通用户。
