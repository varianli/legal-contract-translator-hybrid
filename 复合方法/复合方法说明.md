# 复合方法：Markdown 结构 + Run 格式映射

你的设想是可行的，但需要明确分工：

- Markdown 负责给模型看结构、层级、缩进语义和大段上下文。
- Word run 负责确定性提取粗体、下划线、高亮等格式片段。
- 大模型负责把“原文有格式的中文片段”映射到“英文译文中的对应短语”。
- 程序负责把这些格式加回英文 DOCX，并输出 checklist 防错漏。

## 为什么合理

纯 run 翻译太碎，模型上下文差；纯大段翻译格式无法确定性保留。复合方法把两者拆开：

1. 用 Markdown block 表达合同的段落、标题、条款结构。
2. 用大段 Markdown 调用 API，得到上下文更好的英文。
3. 单独提取 run 里的格式信息。
4. 让模型只做格式映射，不让模型负责维护大量 run 标签。
5. 程序根据映射给英文短语加格式。

## 内置提示词逻辑

程序会把提示词拆成两步：

1. Markdown 翻译提示词：告诉模型 Markdown 已经携带条款层级、缩进、列表和一部分可见结构，但它不是完整的 Word run 格式。模型先在 Markdown 结构下进行大段法律英语翻译，保证上下文准确。
2. Run 格式映射提示词：告诉模型逐 block 检查 run 结构里包含、但 Markdown 不一定表达完整的格式信息，例如粗体、下划线、高亮。

格式映射阶段会按 checklist 逐段推进：

1. 检查当前 block 的 `format_spans`。
2. 记录每个有格式的原文内容和格式类型。
3. 在当前 block 的英文译文中定位对应的英文短语。
4. 返回精确 `target_text`，由程序把格式加回英文 DOCX。
5. 当前 block 全部检查完成后，再进入下一 block。

## 使用方法

1. 双击 `一键启动.bat`。
2. 输入 OpenAI 或 DeepSeek API Key。
3. 选择或拖入 `.docx` 合同。
4. 确认输出目录。
5. 点击“开始复合方法翻译”。
6. 如需提前停止，点击“中止并导出当前进度”。程序会等当前 API 批次返回后立刻导出已完成内容，未完成 block 保留原文。

## 输出文件

默认输出在 `复合方法/输出`：

- `*_复合方法英文翻译.docx`
- `*_source_blocks.md`
- `*_translated_blocks.md`
- `*_复合方法checklist.md`
- `*_复合方法明细.json`

## Checklist 状态

- `APPLIED`：格式已成功应用到英文短语。
- `MISSING_MAPPING`：模型没有返回该格式片段的映射。
- `TARGET_NOT_FOUND`：模型返回了英文短语，但程序没有在译文中精确找到。
- `HAS_CHINESE_IN_TRANSLATION`：译文仍含中文。
- `MISSING_TRANSLATION`：该 block 没有译文。
- `CANCELLED`：用户主动中止，文件已按当前进度导出。

## 重要边界

这个方法保留的是“格式语义”，即把原文的粗体/下划线/高亮加到英文对应短语上；它不会强求 Word 原 run 位置完全不变。对于法律合同，这通常比逐 run 硬替换更自然，也比完全不管格式更可审计。

如果你需要 Word 原 run 的位置和数量完全不变，请用上一层目录中的“并发 run 保格式模式”。
